# Governance Plan Visualizer

## Overview

Transform a governance brief into a beautiful, self-contained interactive HTML dashboard branded for Credo AI. The output is a single `.html` file — no server, no dependencies to install, just open in a browser. Serves two audiences: executives who need a posture summary, and the AI/risk team who need to explore risks, compliance obligations, and controls in depth.

## Paged layout — this report is multi-page

This dashboard renders as a **multi-page document**, not one long scroll: exactly one page is visible at a time, switched via a sticky top nav with prev/next and keyboard paging. Copy the markup and script wholesale from `design/paged-report-shell.html` (locate it via the same glob patterns used for `REPORT_DESIGN.md`) and render it in this skill's **product tokens**, not the shell's marketing tokens. The foundation's "Paged layout" and "Print / PDF" sections are the contract; everything below maps this report's content onto it.

Use the exact paged class/function names from the foundation: page containers `.page` (+ `.is-active`, with `.page--cover` for the dark hero), nav `.report-nav` / `.report-nav__item` (`data-page`, `is-active`) / `.report-nav__counter` / `.report-nav__select` (built from each page's `data-title`), the single `goToPage(id, opts)` navigation function with hash deep-linking, foot `.page-nav` prev/next, Left/Right arrow paging (ignored in inputs), and the focus-pulse class `.is-flash`.

## What to produce

A single self-contained HTML file. Each top-level view below is one `.page` container; exactly one is active at a time. Page order:

1. **Executive Overview (page 1, hero cover)** — the dark `.page--cover` hero treatment: Credo AI logo, system name in Instrument Sans, deployment status / autonomy / domain / jurisdiction pill badges, and a row of headline stat tiles (total risks by tier, controls planned, obligations count) so the artifact opens with the posture at a glance — followed by the executive narrative: narrative framing cards (what this plan is / why it matters now / the core message), system context, risk posture tiles, top actions, compliance snapshot, gaps flag. This is page 1; the dark cover treatment carries the header.
2. **How to read this plan** — short interpretive panel: what severity × likelihood scores mean, how to read the matrix, what the swim lanes imply about sequencing
3. **Risk View** — 5×5 risk matrix + sortable expandable risk table with control linkage
4. **Compliance View** — obligations grouped by source regulation, annotatable checklist
5. **Action View** — swim lane control cards showing mitigated risks and policy requirements
6. **Governance Gaps** — Critical/High risks with no associated controls

The **sticky navigation bar** is the paged `.report-nav` — a DOM-built button per page (from each page's `data-title`), the active page's button marked `is-active`, a `.report-nav__counter` ("N / M"), and a `.report-nav__select` jump menu below 640px. It replaces the old jump-link/scrollspy bar entirely (there is no scrollspy under paging — the active page _is_ the active nav item).

The **footer** ("Generated by Credo AI Governance Hub · [ISO date]") sits below the pages — read `./docs/credoai/org.md` (local) or `~/.claude/credoai/org.md` (global fallback); if either exists, append " · Prepared for [Organization Name]".

Pin the document title in `<head>` to exactly this format:

## Technical approach

Single self-contained HTML file: vanilla JS + inline CSS, no build step, no npm. Start from the paged shell at `design/paged-report-shell.html` — copy its `.page` / `.report-nav` markup and its `goToPage` script verbatim, then restyle in product tokens and drop this report's content into the pages. Tailwind via CDN is fine for layout utilities, but **all color, type, spacing, radius, and shadow values must come from the bundled Credo AI design tokens** (see below) — do not substitute Tailwind defaults.

```html
<script src="https://cdn.tailwindcss.com"></script>
```

The risk matrix is a pure CSS grid — no charting library needed.

### Print / PDF (paged)

The foundation's `@media print` block is the contract — copy it from the shell. It un-hides every `.page` (`display:block !important`), inserts `page-break-before: always` between pages (`avoid` on the first), hides the nav / prev-next / counter / print button, and force-opens collapsibles so nothing is hidden in the PDF. Because the risk table rows and the Quick Wins note sub-rows collapse by default in this report, ensure the print block **expands every expandable row** (give them the `.is-collapsed`-style hook the print CSS un-hides, or add the rows to the print override) so the printed PDF shows full rationale, all "Addressed by" chips, and every Quick Wins note. The result is a complete multi-page PDF, one report view per printed page.

### Design tokens — inline at gen time

The Credo AI design system ships a single CSS variables file (`credo-design-tokens.css`) bundled as an asset in this skill. **Read it at generation time and inline its contents inside `<style>` tags in the `<head>`.** This gives the dashboard the full canonical token set (colors, type scale, spacing, radii, shadows) used across the Credo AI platform — and means every viz output stays in sync with the brand kit when it updates.

### Typography — Inter for product, Instrument Sans for hero

The design system splits type by surface: **Inter** is the product face (used in the platform UI for density and readability at small sizes), **Instrument Sans** is the marketing/brand face. Dashboards are hybrid — they're presentational, but they carry dense data. The convention:

- **Body, tables, cards, pills, buttons, captions:** Inter (`var(--font-sans)`)
- **Hero system name in the dark header banner only:** Instrument Sans (`var(--font-marketing)`) — this is the one brand-marketing moment in the artifact

Both fonts are imported by `credo-design-tokens.css`. Apply globally:

```css
body {
  font-family: var(--font-sans);
  color: var(--grey-800);
  background: var(--ghost-white);
}
.font-marketing {
  font-family: var(--font-marketing);
}
```

**Product type ladder** (from `colors_and_type.css`, derived from the platform Figma):

### Design principles (from the brand guide and the platform Figma)

- **Less is more** — generous whitespace, don't clog the layout. The brand guide repeats this rule four times.
- **Strong text hierarchy via weight, not color** — use 400/500/600/700; never colorize body words. Highlights are bold, not violet.
- **Left-aligned text** — always, except in centered hero moments.
- **Sentence case in body and UI** — no ALL-CAPS for emphasis. The only uppercase usage is the eyebrow/overline (10px / 700 / +0.08em tracking).
- **Rounded corners (product scale)** — `--r-sm` (6px) for buttons, status pills, and inputs; `--r-md` (8px) for small cards and list items; `--r-lg` (12px) for primary content cards; `--r-xl` (16px) for hero panels. Reserve `--r-pill` (999px) for header badges only — the product surface uses 6px pills, not capsules.
- **Sharp corners on data tables** — the brand guide is explicit; don't round table cells.
- **Soft purple-tinted shadows** — never hard black drop-shadows. Use `--shadow-xs` / `--shadow-sm` for cards (the elevation default), `--shadow-md` for hover, `--shadow-lg` for floating menus and modals only.
- **Gradient for movement** — French Violet → Mimi Pink (`linear-gradient(90deg, var(--credo-purple), var(--mimi-pink))`) as a 2–3px accent line at the bottom of the header. Never on backgrounds, never on text.
- **No emoji, no decorative italics, no left-border-accent cards** — these are explicitly banned in the brand guide.

## Data structure

Embed as a JS constant at the top of the `<script>` block:

```javascript
const DATA = {
  system: { name: "...", status: "...", autonomy: "...", domain: "...", jurisdiction: "..." },
  risks: [{ name, type, severity, likelihood, score, tier, rationale }],
  controls: {
    doNow:    [{ label, mitigates: [riskName, ...], policyRequirement }],
    doNext:   [{ label, mitigates: [riskName, ...] }],
    quickWins:[{ label, mitigates: [riskName, ...], note }]
  },
  compliance: [{ text, source }],
  gaps: ["risk name", ...]
};
```

Derive the control→risk reverse lookup at runtime (no new data required):

```javascript
const riskControls = {};
[...DATA.controls.doNow, ...DATA.controls.doNext, ...DATA.controls.quickWins].forEach((c) =>
  (c.mitigates || []).forEach((r) => {
    riskControls[r] = riskControls[r] || [];
    riskControls[r].push({ label: c.label, lane: detectLane(c) });
  }),
);

function detectLane(c) {
  if (DATA.controls.doNow.some((x) => x.label === c.label)) return "doNow";
  if (DATA.controls.doNext.some((x) => x.label === c.label)) return "doNext";
  return "quickWins";
}
```

Lane is used to color "Addressed by" chips in the Risk View:

- `doNow` → Critical tier colors (`#260838` bg, white text)
- `doNext` → High tier colors (`#7B04CA` bg, white text)
- `quickWins` → Medium tier colors (`#C9B6FF` bg, `#161616` text)

## Section 1b: How to read this plan

A compact interpretive panel (per the foundation's how-to-read idiom) between the Executive Overview and the Risk View — three or four numbered cards, `--fs-body` text:

1. **Scores are context-specific** — severity × likelihood (each 1–5) scored for THIS system's deployment context; the same risk scores differently elsewhere. Tiers: Critical 20–25, High 12–19, Medium 6–11, Low 1–5.
2. **How to read the matrix** — severity rises left→right, likelihood top is high; darker purple = more critical. Position, not color hue, carries the meaning.
3. **What the lanes imply** — Do Now items block or mitigate Critical/High risks and anchor compliance obligations; Do Next is sequenced work; Quick Wins are low-cost moves worth batching.
4. **Gaps are the headline** — a Critical/High risk with no control is the most decision-relevant fact in this artifact (only when `DATA.gaps` is non-empty).

Keep it tight — this panel orients a first-time reader in under a minute; it is not a methodology essay.

## Section 2: Risk View

**Risk matrix** — 5-column × 5-row CSS grid. Severity on the X axis (1–5, left to right). Likelihood on the Y axis (5–1, high at top). ## Interactivity

Cross-**page** interactions go through `goToPage(targetPageId, { focus: targetElementId })` — switch the page first, then scroll the focused element into view and pulse it with `.is-flash`. Within-**page** interactions (matrix badge → its own table row, column sort, expand/collapse) stay as plain in-place scroll/expand — they never call `goToPage`.

| Interaction                     | Page scope         | Behavior                                                                                          |
| ------------------------------- | ------------------ | ------------------------------------------------------------------------------------------------- |
| Hover risk badge in matrix      | within Risk View   | Tooltip: name, type, rationale                                                                    |
| Click risk badge in matrix      | within Risk View   | Scroll to + expand the matching row in the risk table (same page; no `goToPage`)                  |
| Expand risk table row           | within Risk View   | Show rationale + "Addressed by" chips                                                             |
| Click control chip on risk row  | Risk → Action View | `goToPage(actionView, { focus: controlCardId })` — switch page, scroll to card, `.is-flash` pulse |
| Click risk chip on control card | Action → Risk View | `goToPage(riskView, { focus: riskRowId })` — switch page, scroll to row, expand it, `.is-flash`   |
| "See all actions →" link        | Overview → Action  | `goToPage(actionView)` — switch to the Action View page                                           |
| "See full checklist →" link     | Overview → Compl.  | `goToPage(complianceView)` — switch to the Compliance View page                                   |
| Compliance checkboxes           | within Compliance  | Ephemeral interactive state                                                                       |
| Click risk table column header  | within Risk View   | Sort ascending; click again for descending (in-place)                                             |
| Click a `.report-nav__item`     | any → that page    | `goToPage(item.dataset.page)`; the active page's nav item carries `is-active` (no scrollspy)      |
| Prev / Next foot controls       | adjacent pages     | `goToPage` to the neighbor; Prev hidden on page 1, Next hidden on the last page                   |
| Left / Right arrow keys         | adjacent pages     | Page back/forward; ignored when focus is in an input/select/textarea/contenteditable              |

## Parsing the governance brief

The brief arrives as Markdown. Extract:

- **Header block**: system name, deployment status, autonomy, domain, jurisdiction
- **Risk table rows**: name, type, severity (int), likelihood (int), score (int), tier, rationale
- **Control roadmap items**: for each swim lane, parse lines of the form `- [label] — mitigates: [Risk A], [Risk B]; [optional note or policy text]`.
  - For Do Now lines: `- [label] — satisfies [policy text]; mitigates: [risks]` — extract `label`, `policyRequirement`, and `mitigates` array.
  - For Do Next lines: `mitigates: [risks]` — extract `mitigates` array.
  - For Quick Wins lines: `mitigates: [risks]; [note text]` — extract `mitigates` array and `note`.
- **Compliance obligations**: each bullet as a string; extract `source` from parentheses or after a dash at the end of the line (e.g., "You must document model training data _(EU AI Act Art. 10)_" → source "EU AI Act Art. 10"). If no source marker, leave `source` empty.
- **Governance gaps**: risk names listed under the Governance Gaps heading
